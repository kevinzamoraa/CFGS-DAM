Guía para la elaboración del diagrama UML:
Esta tarea, "Casos de Uso", es la representación visual de los requisitos funcionales que definimos en la tarea anterior. Se trata de mostrar quién usa la aplicación (actores) y qué puede hacer con ella.

Para cumplir con la rúbrica de "100%", necesitamos definir bien los actores, incluir las relaciones <<include>> (obligatorias) y <<extend>> (opcionales), y detallar los flujos paso a paso.

Aquí tienes la propuesta técnica para Temporis:

Tarea 06: Casos de Uso
1. Identificación de Actores
Usuario (Actor Principal): Persona que utiliza la aplicación para gestionar su tiempo. Puede ser un usuario anónimo (limitado) o autenticado.

Sistema Firebase (Actor Secundario): Proporciona los servicios de autenticación, base de datos y almacenamiento.

Sistema Biométrico del Dispositivo: Hardware/Software del móvil que valida la identidad del usuario.

2. Diagrama de Casos de Uso (Estructura)
Para el diagrama, los casos de uso se agrupan en torno al Usuario.

Casos principales: "Gestionar Temporizadores", "Autenticarse", "Configurar Perfil", "Visualizar Estadísticas".

Relaciones <<include>>: Para "Gestionar Temporizadores" o "Configurar Perfil", el sistema incluye obligatoriamente "Iniciar Sesión".

Relaciones <<extend>>: El caso "Iniciar Sesión" puede extenderse a "Autenticación Biométrica" (es opcional si el usuario lo activa). El caso "Configurar Accesibilidad" extiende la navegación principal.

3. Especificación Detallada de Casos de Uso Críticos
He seleccionado los dos procesos más vitales para el funcionamiento de Temporis:

CU-01: Gestión de Temporizadores (CRUD)
Descripción: Permite al usuario crear y manipular sus contadores.

Actores: Usuario, Firebase Firestore.

Precondiciones: El usuario debe haber iniciado sesión (Auth válido).

Flujo Principal:

El usuario accede a la vista de Temporizadores.

El usuario pulsa el botón "Nuevo Temporizador".

El usuario introduce nombre y minutos.

El sistema valida los datos y los envía a Firestore.

La lista se actualiza en tiempo real.

Postcondiciones: El temporizador queda guardado y es persistente en la nube.

CU-02: Autenticación de Usuario (Multimodal)
Descripción: Acceso seguro al sistema mediante correo o métodos avanzados.

Actores: Usuario, Firebase Auth, Sistema Biométrico.

Precondiciones: La aplicación debe estar instalada y tener conexión a internet.

Flujo Principal:

El usuario abre la aplicación.

El sistema muestra opciones de Login (Email o Google).

El usuario introduce credenciales o selecciona cuenta de Google.

El sistema valida con Firebase.

Flujo Alternativo (Extensión Biométrica):
2a. Si el usuario activó la biometría previamente, el sistema lanza el BiometricPrompt.
2b. El usuario coloca su huella o escanea su rostro.
2c. Si es válido, se salta el paso de introducir contraseña manual.

Postcondiciones: El usuario accede al Dashboard y se genera un token de sesión (caducidad 1h).

CU-03: Ajuste de Accesibilidad Dinámica
Descripción: Personalización de la interfaz en tiempo real.

Actores: Usuario.

Precondiciones: Ninguna (accesible desde el menú de configuración).

Flujo Principal:

El usuario navega a "Accesibilidad".

El usuario desliza el slider de "Tamaño de fuente".

El sistema recalcula el contexto (Configuration) y aplica el tamaño a todos los TextViews.

El usuario activa "Alto contraste".

El sistema cambia la paleta de colores de la UI.

Postcondiciones: Los ajustes se guardan en SharedPreferences para futuras sesiones.
